# Project Memory

## Product Positioning

- 项目名称：`heyue_dental`
- 目标载体：平板优先的口腔门诊展示站
- 当前形态：`Vite + React + TypeScript` 静态前端站点
- 使用方式：
  - GitHub Pages 线上访问
  - PWA 添加到平板主屏幕
  - `dist/` 可作为离线网页包分发

## Current Design Direction

- 整体风格关键词：`简洁 / 专业 / 温馨`
- 避免的状态：
  - 太挤
  - 太乱
  - 像表格
  - 一屏塞满细项
- 当前优化原则：
  - 先分组，再展示项目
  - 先给判断信息，再给细项
  - 用留白和层级替代表格式堆砌
  - 减少陌生图库图带来的廉价感或不真实感
  - 页面首屏文案优先站在用户视角表达，不对用户讲“我们怎么改版”

## Mandatory Frontend Reference Rule

- 这是前端页面相关项目
- 以后只要涉及页面优化、结构调整、视觉改版、文案排布修改，都必须优先参考以下目标：
  - 避免：`太挤 / 太乱 / 像表格`
  - 改成：`有分类 / 有重点 / 有留白`
  - 风格参考：`简洁 / 专业 / 温馨`
- 这条规则适用于：
  - 首页
  - 服务项目页
  - 服务详情页
  - 关于我们
  - 专家页
  - 案例页
  - 未来新增页面
- 如果出现信息很多的页面，优先做法不是继续堆内容，而是：
  - 先拆分类
  - 再做摘要
  - 最后把完整明细下沉到详情层

## Information Architecture Rules

- 服务项目页当前按 10 个业务模块展示，不再使用旧的“就诊意图 4 分组”口径
- 一级层是业务模块，二级层是 `sections`
- 每个项目优先展示：
  - 适合谁先看
  - 这类重点
  - 一句话概括
- 总览页不平铺所有明细：
  - 先展示模块和二级分类数量
  - 详情页再按二级分类展开条目

## Service Classification Memory

- 服务项目必须尽量完整覆盖，不允许只保留少量“展示型项目”而丢失业务全貌
- 以后整理 `services.json`、导入脚本、服务项目页面时，优先按 **口腔诊疗业务模块** 分类，而不是按字面零散摆放
- 当前记忆中的推荐一级分类为 10 大类：
  - 挂号 / 诊查 / 基础检查
  - 牙周治疗
  - 牙槽外科 / 拔牙 / 口腔外科
  - 牙体牙髓 / 根管治疗
  - 预防保健 / 儿童口腔
  - 修复固定类
  - 义齿 / 活动修复 / 加工维修
  - 正畸检查 / 正畸复诊 / 正畸辅助项目
  - 正畸治疗
  - 口腔黏膜 / 激光 / 其他处置

## Service Data Structure Rule

- 后续整理服务项目表或导入脚本时，建议至少保留：
  - `项目名称`
  - `一级分类`
  - `二级分类`
- 页面展示可以不把所有字段都直接露出，但数据结构应尽量支持这两级分类
- 典型示例：
  - 普通门诊诊查费 -> 检查诊查 / 门诊诊查
  - 前牙拔除术 -> 口腔外科 / 拔牙
  - 简单充填术 -> 牙体牙髓 / 充填
  - 根管充填术 -> 牙体牙髓 / 根管治疗
  - 洁治 -> 牙周治疗 / 洁牙
  - 固定桥 -> 修复 / 固定修复
  - 塑料可摘局部义齿 -> 修复 / 活动义齿
  - 替牙期安氏I类错颌固定矫治器正畸治疗 -> 正畸治疗 / 替牙期固定矫治
  - 正畸保持器治疗 -> 正畸治疗 / 保持

## Service Classification Edge Cases

- 以下项目容易被分错，后续分类时必须统一口径：
  - `冠周炎局部治疗`：统一放 `牙周治疗` 或 `口腔外科`，不能来回漂移
  - `牙开窗助萌术 / 正畸牙开窗术`：更适合放 `口腔外科`
  - `固定矫治器复诊处置`：放 `正畸辅助项目`，不要混入正式正畸疗程
  - `固定桥 / 粘结桥`：放 `固定修复`
  - `塑料可摘局部义齿 / 即刻义齿 / 附着体义齿`：放 `活动义齿修复`
  - `CT / 曲面断层 / RVG`：放 `检查影像`

## Ongoing Update Rule

- 从现在开始，每次做页面优化或信息架构调整时，必须同步更新本文件
- 更新内容至少包括：
  - 本次改了什么
  - 为什么改
  - 当前页面风格或结构规则有没有变化
  - 是否影响部署、PWA 或内容展示策略
- 从现在开始，每次完成代码修改后，默认必须执行一次打包：
  - 命令：`npm run build`
  - 目的：确保 `dist/` 始终是最新可发布版本，便于同步到 GitHub Pages
  - 这条规则适用于页面改动、服务项目数据改动、导入脚本改动，以及任何会影响线上展示结果的更新
- 从现在开始，每次“更新 / 优化 / 修改”完成后，默认执行一次 GitHub Pages 同步：
  - 命令：`npm run pages:sync`
  - 适用范围：页面优化、文案修改、服务项目调整、数据导入结果变更、样式修改、结构调整
  - 目标：保证本地改动、`dist/` 产物、GitHub 仓库、GitHub Pages 保持一致
- 完成代码修改后，默认一起提交并 push，保持仓库状态和 Memory 同步
